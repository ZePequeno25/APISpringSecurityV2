📌  Sobre este Tutorial
Neste guia, você aprenderá a proteger uma API REST Spring Boot utilizando Spring Security + JWT. Você vai
construir, passo a passo, um sistema completo de autenticação e autorização com papéis (roles), pronto para
ser usado em aplicações reais.

package com.exemplo.secrest.security.service;
import com.exemplo.secrest.entity.User;
import lombok.Getter;
import org.springframework.security.core.GrantedAuthority;
import org.springframework.security.core.authority.SimpleGrantedAuthority;
import org.springframework.security.core.userdetails.UserDetails;
import java.util.Collection;
@Getter
public class UserDetailsImpl implements UserDetails {
    private final User user;
    public UserDetailsImpl(User user) {
        this.user = user;
    }
    @Override
    public Collection<? extends GrantedAuthority> getAuthorities() {
        return user.getRoles()
                .stream()
                .map(role -> new SimpleGrantedAuthority(role.getName().name()))
                .toList();
    }
    @Override
    public String getPassword() {
        return user.getPassword();
    }
    @Override
    public String getUsername() {
        return user.getEmail();
    }
    // Os métodos abaixo retornam true para simplificar o exemplo
    @Override public boolean isAccountNonExpired() { return true; }
    @Override public boolean isAccountNonLocked() { return true; }
    @Override public boolean isCredentialsNonExpired() { return true; }
    @Override public boolean isEnabled() { return true; }
}
Explicação: O Spring Security exige que as informações do usuário estejam encapsuladas em um
objeto UserDetails. Aqui, convertemos nossa entidade User para esse formato. O método
getAuthorities() transforma cada Role em uma GrantedAuthority, que o Spring Security usa para
tomar decisões de autorização.

public class JwtTokenService {
    // ⚠  Em produção, JAMAIS hardcode secrets. Use variáveis de ambiente.
    private static final String SECRET_KEY = "4Z^XrroxR@dWxqf$mTTKwW$!@#qGr4P";
    private static final String ISSUER = "secrest-api";
    public String generateToken(UserDetailsImpl user) {
        try {
            Algorithm algorithm = Algorithm.HMAC256(SECRET_KEY);
            return JWT.create()
                    .withIssuer(ISSUER)
                    .withIssuedAt(creationDate())
                    .withExpiresAt(expirationDate())
                    .withSubject(user.getUsername())
                    .sign(algorithm);
        } catch (JWTCreationException e) {
            throw new JWTCreationException("Erro ao gerar token JWT", e);
        }
    }
    public String getSubjectFromToken(String token) {
        try {
            Algorithm algorithm = Algorithm.HMAC256(SECRET_KEY);
            return JWT.require(algorithm)
                    .withIssuer(ISSUER)
                    .build()
                    .verify(token)
                    .getSubject();
        } catch (JWTVerificationException e) {
            throw new JWTVerificationException("Token JWT inválido ou expirado");
        }
    }
    private Instant creationDate() {
        return ZonedDateTime.now(ZoneId.of("America/Sao_Paulo")).toInstant();
    }
    private Instant expirationDate() {
        return 
ZonedDateTime.now(ZoneId.of("America/Sao_Paulo")).plusHours(4).toInstant();
    }
}
Explicação: Esse serviço concentra as operações JWT. generateToken cria um token com emissor,
data de criação, expiração e o email do usuário como "subject". O token é assinado com HMAC256.
getSubjectFromToken verifica a assinatura e extrai o email (subject). Em produção, a chave secreta
deve vir de variável de ambiente, não ficar hardcoded.

Praticar a injeção do UserDetails no controller.
Requisitos
1. O endpoint deve retornar o id, email e lista de roles do usuário autenticado.
2. Deve ser acessível apenas para usuários com qualquer papel (qualquer token válido).
3. Formato de resposta sugerido:
{
    "id": 1,
    "email": "maria@email.com",
    "roles": ["ROLE_CUSTOMER"]
}
Passos sugeridos
1. Criar um DTO de resposta UserProfileDto dentro do pacote dto:
public record UserProfileDto(Long id, String email, List<String> roles) {}
2. No UserController, adicione o novo endpoint:
@GetMapping("/me")
public ResponseEntity<UserProfileDto> getCurrentUser(Authentication 
authentication) {
    // O Spring Security já injeta o objeto Authentication automaticamente
    String email = authentication.getName();
    User user = userRepository.findByEmail(email)
            .orElseThrow(() -> new RuntimeException("Usuário não encontrado"));
    List<String> roles = user.getRoles().stream()
            .map(role -> role.getName().name())
            .toList();
    return ResponseEntity.ok(new UserProfileDto(user.getId(), user.getEmail(), 
roles));
}
3. Injete o UserRepository no controller (ou faça via UserService – escolha a melhor prática).
4. Configure a segurança no SecurityConfiguration para permitir acesso autenticado ao novo
endpoint:
.requestMatchers("/users/me").authenticated()
5. Teste com um token válido: GET /users/me deve retornar os dados do usuário.
